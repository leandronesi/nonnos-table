Cosa resta, tolta la fuffa

Niente framework, niente "agentic". Una cosa sola.

L'industria intorno agli agent ha prodotto un sacco di parole: "agentic framework", "orchestration layer", "tool calling architecture", "multi-agent system", "ReAct", "CoT", "MCP". La verita' sotto e' molto piu' piccola di tutto questo.

Un agente = un LLM + una cartella di istruzioni

Punto. Tutto il resto sono parole.

Quando un agent funziona bene, e' perche' la cartella delle istruzioni e' organizzata bene. Quando funziona male, e' perche' la cartella e' un casino. Non e' piu' complicato di cosi'.

Nelle prossime tre slide ti faccio vedere com'e' fatta dentro quella cartella. Tre tipi di file, livelli di profondita' diversi, stesso principio.

La macroistruzione — il file "fuori"

Un solo file, in cima. Dice al modello chi e', cosa fa, dove andare a cercare il resto.

Apri la cartella di qualunque agente serio e troverai, alla radice, un file che si chiama tipicamente CLAUDE.md (per Claude Code), .cursorrules (per Cursor), o AGENT.md. E' la macroistruzione: il primo file che il modello legge appena parte una sessione.

Cosa contiene? Solo tre cose:

  • Identita'. Chi sei. Su quale progetto stai lavorando. Con quale tono.
  • Vincoli. Le poche regole che valgono sempre, indipendentemente dal task.
  • Mappa. Dove vivono le istruzioni piu' specifiche. "Per X vai in spec/. Per Y vai in skills/."

E' come quando entri in un'azienda nuova e ti danno il "manualetto del nuovo arrivato": chi siamo, cosa facciamo, e dove trovare i documenti specifici.

Esempio concreto. Il CLAUDE.md del deck che stai guardando — sta in docs/deck/CLAUDE.md — e' lungo 80 righe. Dentro c'e' la palette, la voce, dove vivono le slide. Quando un agent lavora sul deck, lo legge per primo. Tutto qui.

Le sub-istruzioni — le cartelle dentro

Il file root rimanda a sotto-cartelle. Ognuna ha file specifici, caricati solo quando serve.

Non e' realistico avere TUTTO dentro CLAUDE.md. Sarebbe lungo 5000 righe, illeggibile, e il modello si perderebbe nel rumore. Quindi la macroistruzione delega: "per la voce vai in spec/voice.md. Per i requisiti di prodotto, in spec/product.md. Per chi e' il lettore, in spec/audience.md."

docs/deck/ ├── CLAUDE.md # macroistruzione (il "fuori") ├── spec/ # sub-istruzioni di contesto │ ├── product.md # cos'e' il deck │ ├── audience.md # chi e' Mimmo │ └── voice.md # tono, regole copy ├── content/ # il contenuto vero └── ...

Il modello, quando il task richiede contesto specifico, va a leggere quei file. Quando non serve, li ignora. E' navigazione, non ricerca — il modello sa dove andare perche' la macroistruzione glielo ha detto.

La cosa interessante: questa struttura non e' un'invenzione tecnica. E' una biblioteca. Indice in cima, sezioni dentro, schede piu' dettagliate ancora dentro. Cervello umano e cervello LLM navigano informazioni nello stesso modo.

Le skill — istruzioni di comportamento

Caso speciale di sub-istruzione: come comportarsi in una situazione specifica.

A volte non basta dire al modello "che voce usare" o "cos'e' il prodotto". Devi spiegargli come comportarsi davanti a un certo tipo di task. Queste istruzioni speciali si chiamano skill.

Una skill e' un file (di solito SKILL.md) dentro una sotto-cartella dedicata. Contiene:

  • Quando si attiva (in linguaggio naturale, non keyword)
  • Cosa deve fare il modello in quella situazione
  • Esempi buoni e cattivi

Il modello carica la skill solo quando il contesto la richiede. Le altre volte non occupa spazio.

Esempio concreto. Il visualizer cinematografico che hai visto a slide 9 e 11 e' stato costruito seguendo una skill chiamata motion-reel, in docs/deck/skills/motion-reel/SKILL.md. Dentro c'e' scritto: "quando devi animare il visualizer del deck, usa particelle vere, trail veri, glow pulsante, niente transition CSS basic". Il modello, quando ha scritto le animazioni, ha letto quel file e si e' comportato di conseguenza.

Quindi: la skill e' una sub-istruzione un po' piu' specializzata. Stesso pattern di prima, livello diverso.
